# DC-145: [EPIC] Simplified Companion (Candy) — Exploratory

**Status:** backlog  
**Priority:** high  
**Type:** spike  
**Wiki:** [[projects/DaemonCraft/notes/simplified-companion-research]]  

## Goal

Strip complexity from our current architecture while keeping the sound single-LLM foundation. Create a new bot template — **Simplified Companion** — for a bot named **Candy** that uses the best primitives from all three forks (ours, Fede's, Pablo's) with the simplest effective prompting.

## Core Hypothesis

> A simpler, clearer prompt with the same primitives will perform better than a complex prompt with recovery heuristics layered on top.

Our current SOUL-minecraft.md is 345 lines with DC-112-specific "trust heartbeat" logic. Fede's is 178 lines with a simpler "perceive, think, act, observe" loop. The primitives are **identical** across all repos. The difference is **how we instruct the LLM to use them**.

## What We Keep (Our Architecture)

| Component | Why |
|-----------|-----|
| Single-LLM (gateway owns cognition) | Memory, context, cross-platform comms, persistent sessions |
| Hermes profiles | Config inheritance, isolation, memory persistence |
| bg_* tools for long tasks | Player doesn't wait for 64-log collection |
| Foreground tools for immediate tasks | Chat, place one block, respond to player |
| Heartbeat every 30s | Keeps context fresh, reports world state |
| Wake-up on chat/mention | Responsive when player needs us |
| `allowSprinting=false` | Already fixed, prevents pathfinder step bug |

## What We Scrap (Complexity)

| Complexity | Why |
|------------|-----|
| "Trust heartbeat, don't perceive" conditional logic | LLM is better at deciding when to perceive |
| DC-112-specific prompting labels | Architecture labels confuse the model |
| "Multi-action turns" (DC-134) | LLMs are bad at batching; one tool at a time is clearer |
| Complex stuck-recovery heuristics | "Try something different after 3 fails" is enough |
| BODY.md / SOUL.md separation | We want ONE conversation for talk+work |
| "Your turn ends immediately" framing | Model doesn't need to know turn boundaries |

## Notes

**Why "Candy"?** Because it's a sweet, simple name for the simplest effective companion. No elaborate persona, no complex backstory. Just a helpful bot that can talk and work.

**Why a spike epic?** Because this is experimental. If Candy works better than the current companion, we migrate all casts to the simplified template. If not, we learn what matters and what doesn't.

**A/B testing:** Run Candy alongside the current companion cast. Compare:
- Task completion rate
- Time to complete simple tasks (build shelter, collect 16 logs)
- Number of stuck events per hour
- Token usage per task
- Player satisfaction (subjective)
