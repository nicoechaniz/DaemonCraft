# DC-137: [EPIC] Embodied Body Intelligence — From Stuck Zombie to Competent Agent

**Status:** backlog
**Priority:** high
**Type:** epic
**Parent:** none
**Wiki:** [[projects/DaemonCraft/notes/embodiment-gap-analysis]]

## Goal

Close the embodiment gap: build a pluggable body intelligence layer so Minecraft agents can execute physical tasks autonomously without getting stuck, burning tokens on replanning, or waiting 30s for the LLM to notice a problem.

## Problem Statement

Mineflayer pathfinder cannot execute autonomously for more than a few seconds. The current architecture invested heavily in the "brain" (gateway sessions, heartbeat injection, event classification) but left the "body" completely unchanged. Result: agents plan elaborate goals, then get stuck on step 3 and stay there like dead zombies.

The old hermescraft archive proves this: background task primitives (`mc_bg_goto`, `mc_bg_collect`, etc.) existed but were **all marked "NOT TESTED"** because the body layer couldn't execute reliably enough to warrant testing.

## Proposed Architecture

### Phase 1: Stop the Bleeding (immediate)
- Cap `max_iterations` and `turn_timeout` per gateway turn
- Prevents 15-minute zombie turns where the bot is stuck and the LLM is still "thinking"

### Phase 2: BodyController Interface
- Define `BodyController` abstraction:
  - `execute(task: TaskPlan) -> ExecutionReport`
  - `cancel() -> void`
  - `getStatus() -> BodyState`
  - Event emitter: `stuck`, `recovered`, `completed`, `needs_help`
- Wrap existing mineflayer behavior in `MineflayerBody` implementing the interface
- No behavior change yet — just clean abstraction

### Phase 3: Body-Side Recovery Behaviors
- Add stuck detection to `MineflayerBody` (position unchanged for 10s + pathfinder active)
- Implement 2-3 recovery strategies:
  1. Jump + retry current path
  2. Reposition 1-2 blocks and replan
  3. Switch to alternate path (avoid vs. break obstacle)
- Only escalate `needs_help` to brain when all recovery strategies fail
- Brain receives `TaskPlan` at goal level, not micromanaged block placements

### Phase 4: Brain-Body Separation
- Brain (gateway/LLM) issues `TaskPlan` containing goal + constraints
- Body executes autonomously, reports progress via events
- Brain intervenes only on `needs_help` or when goal changes
- Chat/social behavior stays in brain; movement/manipulation stays in body

### Phase 5: Voyager-Style Training Pipeline
- Integrate Fede's training logger (`agents/training/logger.py`) into main repo
- Log every tool execution: success, failure, recovery, final outcome
- Filter for "successful recovery from stuck" examples
- Convert to HuggingFace dataset (`training/convert_dataset.py`)
- Fine-tune small model (Qwen/Axolotl) on body-intelligence task
- Future: `AndyBody` drop-in replacement using fine-tuned model

### Phase 6: Andy Bot Integration (future)
- `AndyBody` implements `BodyController`
- Uses Baritone, learned policies, or hybrid controllers
- Drops in without changing brain code
- Benchmark: build a 5x5 house autonomously without getting stuck

## Notes

**Why not just fix mineflayer-pathfinder?** Because pathfinder is a general-purpose library. Our body layer needs Minecraft-specific smarts: knowing when to jump, when to break vs. go around, how to recover from being pushed by a mob, how to place scaffolding. These are policy decisions, not graph search optimizations.

**Why a separate BodyController instead of more skills?** Skills guide the brain's decisions. They don't help when the body is physically stuck. The gap is at the execution layer, not the planning layer.

**Why Voyager-style training?** Because hand-crafting recovery heuristics for every stuck scenario is impossible. Logging real executions and learning from successful recoveries scales better.
